01

Architecture globale

LazyPO est une Single Page Application sans framework JavaScript. La navigation est gérée par un routeur maison (sidebar.js) qui swape le contenu de la page sans rechargement complet, préservant l'état des modules persistants (Focus FM, session, realtime).

graph TB subgraph CLIENT["🖥️ Navigateur (Client)"] direction TB SB["sidebar.js\nRouter SPA"] AUTH["auth.js\nAuthentification"] SESS["session.js\nGestion session"] FM["focusfm.js\nSpotify Player"] DEMO["demo.js\nDonnées fictives"] subgraph PAGES["Pages / Outils"] P1["🏠 index.html"] P2["✉️ lazypo_generator.html"] P3["📋 sprintplanner.html"] P4["🔍 jirarepo.html"] P5["📝 livenote.html"] P6["🧠 quiz.html"] P7["👤 account.html"] end end subgraph BACKEND["☁️ Supabase (Backend as a Service)"] direction LR SBAUTH["Auth\n(PKCE/email)"] SBDB["PostgreSQL\n5 tables"] SBREALTIME["Realtime\n(WebSocket)"] SBSTORAGE["Storage\n(avatars, livenote-temp)"] end subgraph EXTERNAL["🌐 Services Externes"] SPOTIFY["Spotify API\nWeb Playback SDK"] GFONTS["Google Fonts\nDM Sans / DM Mono"] CDN["CDN (jsDelivr)\nXLSX · PptxGenJS · CodeMirror"] end SB -->|"swap HTML\n+ re-run scripts"| PAGES AUTH --> SBAUTH SESS --> SBDB FM --> SPOTIFY P5 --> SBREALTIME P6 --> SBDB P7 --> SBSTORAGE CLIENT --> GFONTS CLIENT --> CDN
Fig. 1 — Vue d'ensemble de l'architecture LazyPO
Principe clé : Aucun build step, aucun framework (React, Vue…). Chaque outil est un fichier HTML autonome qui inclut ses scripts inline. Le routeur SPA évite les rechargements de page tout en maintenant la cohérence de l'état.

Authentification — auth.js

L'authentification utilise Supabase Auth avec email/password + un système d'approbation manuelle par un admin. Sans approbation, l'utilisateur voit une page d'attente.

Gestion de l'avatar

Les avatars sont uploadés dans le bucket Supabase Storage avatars avec le chemin {user_id}/avatar.{ext}. Le bucket est en lecture publique, écriture authentifiée.

Algorithme SM-2 — Répétition espacée

L'algorithme SuperMemo 2 calcule l'intervalle optimal entre deux révisions d'un mot, selon la qualité de la réponse de l'utilisateur.

flowchart TD ANS["Réponse utilisateur"] --> Q{"Qualité de réponse\nq = ?"} Q -->|"Correct du premier coup\nq = 5"| Q5["q = 5"] Q -->|"Correct après hésitation\nq = 3"| Q3["q = 3"] Q -->|"Incorrect\nq = 1"| Q1["q = 1"] Q5 --> EF_CALC["Nouveau EF =\nEF + (0.1 - (5-q) × (0.08 + (5-q) × 0.02))"] Q3 --> EF_CALC Q1 --> EF_RESET["EF inchangé\nInterval = 1 (reset)"] EF_CALC --> EF_CLAMP["EF = max(1.3, EF)"] EF_CLAMP --> INT_CALC{"n (répétition) = ?"} INT_CALC -->|"n = 1"| I1["interval = 1 jour"] INT_CALC -->|"n = 2"| I2["interval = 6 jours"] INT_CALC -->|"n > 2"| IN["interval = prev_interval × EF"] I1 --> NEXT["📅 Prochaine révision =\nnow + interval_days"] I2 --> NEXT IN --> NEXT EF_RESET --> NEXT NEXT --> DB["Upsert quiz_progress\n{ ease_factor, interval_days, last_tested }"] style NEXT fill:#0c1e2a,stroke:#3b82f6,color:#60a5fa style DB fill:#0f2a1a,stroke:#34d399,color:#34d399
Fig. 10 — Algorithme SM-2 (SuperMemo 2) de répétition espacée

Tolérance aux fautes de frappe

Les réponses avec 1 caractère de différence (distance de Levenshtein ≤ 1) sont acceptées comme correctes, mais avec un score de qualité légèrement réduit. Cela évite les faux négatifs sur des erreurs de saisie mineures.

Stockage des images

Les images collées ou uploadées dans l'éditeur HTML sont automatiquement envoyées dans le bucket livenote-temp avec un TTL de 30 minutes. Un setTimeout côté client supprime le fichier après expiration.
